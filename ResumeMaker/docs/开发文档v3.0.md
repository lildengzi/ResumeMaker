# 智能简历生成器 - 开发文档 v3.0

> 版本：3.0  
> 更新日期：2026-05-14  
> 文档目标：在不新增功能范围的前提下，整理当前项目的具体技术实现方案，并明确当前高风险实现的收敛方向。  
> 适用范围：当前 Streamlit 学习项目，包含简历编辑、AI 优化、HTML 预览、Markdown 导出、PDF 导出、本地保存。

---

## 1. 版本定位

v1 文档提出了较多未来扩展方向，例如 FastAPI、LangGraph、OCR、网页抓取、评分 Agent、Docker 等。

v2 文档明确了模块化简历模型、样式定制、智能体优化、HTML/PDF/Markdown 输出等核心产品形态。

v3.0 不再扩大功能边界，而是把已有能力整理为一套可维护的技术实现规范：

- 使用 Streamlit 作为唯一应用入口。
- 使用统一 JSON 文档作为前端编辑、预览、导出、AI 优化的唯一事实源。
- 简历内容最终只允许 `basics + modules` 两个内容源；旧顶层字段只用于导入迁移，不参与运行时同步。
- 使用观察者模式同步预览版本、Markdown、PDF 缓存失效等派生结果。
- 页面预览不得被 PDF 生成阻塞；PDF 只在用户明确点击“生成 PDF”后创建。
- 运行中以 `st.session_state.resume_document` 作为工作副本，普通输入框编辑不立即写磁盘。
- 保留当前顺序工作流，不引入正式 LangGraph。
- 保留当前 PDF 截图转 PDF 方案，不切换到文本 PDF。
- 保留 tools 层占位能力，不接入真实 OCR 或生产级网页抓取。

## 2. 总体架构

```text
app.py
  |
  |-- ui/sidebar.py              # 左侧编辑区：简历内容、样式、JD、文件上传
  |-- ui/preview.py              # 右侧 HTML 预览容器
  |
  |-- core/resume_document.py    # 统一 JSON 文档 + 观察者同步
  |-- core/document_commands.py  # 写入命令层：更新字段、模块操作、AI 回写、保存快照
  |-- core/data.py               # 数据归一化、默认数据、旧字段兼容、保存读取
  |-- core/service.py            # AI 优化入口、本地兜底
  |
  |-- agents/
  |     |-- info_collector.py     # 输入整理
  |     |-- resume_writer.py      # LLM 简历优化
  |     |-- factory.py            # Agent 工厂
  |
  |-- workflow/graph.py          # 顺序工作流：InfoCollector -> ResumeWriter
  |
  |-- renderers/
        |-- html_renderer.py      # HTML 预览
        |-- markdown_renderer.py  # Markdown 导出入口
        |-- pdf_renderer.py       # HTML 截图 -> Pillow PDF
```

核心原则：

- Sidebar 只负责修改统一文档。
- Preview 只读取统一文档。
- Markdown 和 PDF 都是统一文档的派生结果。
- AI 优化输出必须回写统一文档。
- 保存到磁盘时只保存稳定的简历数据，不保存 PDF bytes 等运行时缓存。

## 3. 统一 JSON 文档设计

当前项目的核心状态为 `resume_document`。

```json
{
  "resume": {
    "basics": {},
    "modules": []
  },
  "style": {},
  "inputs": {
    "jd_text": "",
    "uploaded_files": []
  },
  "runtime": {
    "preview_version": 0,
    "preview_html": "",
    "preview_signature": "",
    "preview_scale": 100,
    "dirty": false,
    "workflow_logs": []
  },
  "exports": {
    "markdown": "",
    "pdf_bytes": null,
    "pdf_ready_for_version": -1
  }
}
```

### 3.1 字段职责

| 字段 | 说明 | 是否持久化 |
|------|------|------------|
| `resume` | 简历主体数据，只包含 `basics` 和 `modules` | 是 |
| `style` | 当前样式参数，供预览和导出消费 | 是，写入 `config.json` 或 `resume.style` |
| `inputs.jd_text` | 当前 JD 文本 | 否，运行态输入 |
| `inputs.uploaded_files` | 上传文件元信息 | 否，运行态输入 |
| `runtime.preview_version` | 预览版本号，用于缓存一致性 | 否 |
| `runtime.preview_html` | 当前预览 HTML 缓存 | 否，可重新生成 |
| `runtime.preview_signature` | 文档签名，用于判断是否变化 | 否 |
| `runtime.preview_scale` | 预览缩放比例 | 否 |
| `runtime.dirty` | 当前工作副本是否有未保存修改 | 否 |
| `runtime.workflow_logs` | 工作流日志 | 否 |
| `exports.markdown` | 当前 Markdown 派生结果 | 否，可重新生成 |
| `exports.pdf_bytes` | PDF 二进制缓存 | 否，可重新生成 |
| `exports.pdf_ready_for_version` | PDF 对应的预览版本 | 否 |

### 3.2 内容源约束

正式目标结构只保留：

```json
{
  "basics": {},
  "modules": []
}
```

以下旧字段只能在导入迁移阶段读取：

- `education`
- `skills`
- `projects`
- `companyExperience`
- `campusExperience`
- `selfEvaluation`

运行时禁止再次把旧字段同步回 `modules`。原因是双源设计会带来覆盖风险：

```text
用户编辑 modules
  -> 旧字段仍是旧值
  -> ensure_resume_shape() 把旧字段同步回 modules
  -> 用户刚改的内容被覆盖
```

因此 v3.0 的数据方向是：

```text
导入旧数据时：
旧顶层字段 -> modules

运行时：
basics + modules -> UI / Preview / AI / Markdown / PDF

导出或保存时：
不再生成旧顶层字段
```

过渡期如果为了兼容测试或旧函数仍需保留旧字段，必须满足：

- 旧字段只由 `modules` 单向派生。
- 旧字段不得再反向覆盖 `modules`。
- 渲染器、AI、保存逻辑不得优先读取旧字段。

### 3.3 模块模型

简历内容统一使用 `modules` 表达：

```json
{
  "id": "projects_1",
  "title": "项目经历",
  "type": "projects",
  "visible": true,
  "order": 3,
  "content": {
    "items": []
  }
}
```

内置类型：

- `education`
- `skills`
- `projects`
- `companyExperience`
- `campusExperience`
- `selfEvaluation`
- `custom`

新增模块统一使用 `custom`，用户通过标题和内容表达具体用途。

### 3.4 模块内容判定

渲染前必须判断模块是否有真实内容。只判断 `visible` 不够。

建议实现：

```python
def has_module_content(module: dict) -> bool:
    content = module.get("content", {})
    items = content.get("items", [])
    if not isinstance(items, list):
        return False

    for item in items:
        if isinstance(item, str) and item.strip():
            return True
        if isinstance(item, dict):
            for value in item.values():
                if isinstance(value, str) and value.strip():
                    return True
                if isinstance(value, list) and any(str(v).strip() for v in value):
                    return True
    return False
```

HTML、Markdown、PDF 必须统一使用：

```python
visible_modules = [
    module for module in modules
    if module.get("visible", True) and has_module_content(module)
]
```

正式导出中不应出现“暂无项目经历”“未填写技能”等空模块占位。

## 4. 观察者模式实现

观察者实现位于：

```text
core/resume_document.py
```

### 4.1 核心 API

```python
create_resume_document(...)
ensure_resume_document(session_state)
get_resume_document(session_state)
get_document_runtime(session_state)
get_document_exports(session_state)
sync_session_aliases(session_state, include_widget_keys=False)
notify_document_changed(session_state, event="change")
```

### 4.2 观察者列表

```python
DOCUMENT_OBSERVERS = [
    _normalize_document_observer,
    _preview_observer,
    _markdown_observer,
    _pdf_invalidation_observer,
    _legacy_alias_observer,
]
```

执行顺序必须保持：

1. 归一化文档。
2. 计算预览签名，判断文档是否变化。
3. 文档变化时更新 Markdown。
4. 文档变化时清空 PDF 缓存。
5. 同步兼容旧代码仍使用的 session key。

预览 HTML 必须按签名缓存：

```python
signature = json.dumps(
    {
        "resume": document["resume"],
        "style": document["style"],
        "preview_scale": document["runtime"]["preview_scale"],
    },
    ensure_ascii=False,
    sort_keys=True,
)
```

只有签名变化时才重新调用 `build_full_preview_document()`，结果写入：

```python
document["runtime"]["preview_html"]
```

### 4.3 Streamlit widget key 约束

Streamlit 不允许在 widget 实例化后修改其对应的 `st.session_state` key。

因此：

- `jd_input`、`preview_scale` 这类 widget key 只能在初始化阶段回写。
- 观察者运行时只能从 widget key 读取值，并写入 `resume_document`。
- `sync_session_aliases(..., include_widget_keys=True)` 只能在 widget 创建前调用。
- 普通观察者同步必须使用默认值 `include_widget_keys=False`。

## 5. 页面状态流

### 5.1 初始化

`app.py:init_state()`

流程：

1. 从 `data/resume_data.json` 加载简历。
2. 调用 `ensure_style_state()` 初始化样式状态。
3. 创建 `resume_document`。
4. 将兼容 session key 指向统一文档。
5. 初始化历史快照、上传控件 key、提示信息等运行态字段。

### 5.2 Sidebar 编辑

`render_sidebar()` 调用：

```text
render_style_controls()
render_basics_editor()
render_modules_editor()
render_jd_and_file_controls()
notify_document_changed()
```

技术要求：

- 编辑器可以继续读写 `st.session_state.resume_data`。
- 因为 `resume_data` 是 `resume_document["resume"]` 的兼容引用，所以实际修改仍落在统一文档中。
- Sidebar 渲染完成后调用 `notify_document_changed()`。
- 观察者负责生成 Markdown、清空 PDF 缓存、更新预览版本。
- 普通输入框编辑只更新内存工作副本，不立即写 `data/resume_data.json`。
- 只有用户点击“保存当前内容”、AI 生成完成、上传头像、增删移动模块等明确动作后才写磁盘。

### 5.3 Preview 渲染

`render_main_panel()` 从统一文档读取：

```python
document = get_resume_document(st.session_state)
render_resume_preview(
    resume_data=document["resume"],
    style_params=get_current_style_params(),
    preview_version=document["runtime"]["preview_version"],
    workflow_logs=document["runtime"]["workflow_logs"],
)
```

`ui/preview.py` 不负责状态变更，只负责调用：

```python
components.html(document["runtime"]["preview_html"], ...)
```

如果 `preview_html` 为空，才允许临时调用 `build_full_preview_document()` 作为兜底。

### 5.4 PDF 导出

PDF 生成入口：

```python
render_resume_document_pdf(resume_document)
```

流程：

```text
resume_document
  -> build_pdf_html_document_from_resume_document()
  -> build_pdf_html_document()
  -> build_full_preview_document()
  -> Playwright 截取 .resume-a4
  -> Pillow 转 PDF
```

PDF 缓存规则：

- `exports.pdf_ready_for_version == runtime.preview_version` 时复用缓存。
- 文档变化后观察者将 `exports.pdf_bytes` 置空。
- 下载按钮不得调用 `ensure_pdf_ready()` 触发重生成。
- 用户必须先点击“生成 PDF”，生成成功后才显示“下载 PDF 简历”。

推荐 UI：

```python
if st.button("生成 PDF"):
    prepare_pdf()

pdf_bytes = document["exports"].get("pdf_bytes")
if pdf_bytes:
    st.download_button("下载 PDF 简历", data=pdf_bytes, ...)
```

## 8. 持久化策略

跨会话唯一来源是：

```text
data/resume_data.json
```

运行时工作副本是：

```text
st.session_state.resume_document
```

数据流：

```text
data/resume_data.json
        -> 启动时读取一次
st.session_state.resume_document
        -> 侧边栏手动修改 / 智能体回写
内存中的 resume_document
        -> 保存按钮或明确动作后写回 JSON
data/resume_data.json
```

禁止在每次 Streamlit rerun 末尾自动保存。

### 8.1 推荐改造顺序

第一步：修改 `ensure_resume_shape()`，只在检测到没有 `modules` 时读取旧顶层字段。

第二步：如果数据已经有 `modules`，直接以 `modules` 为准，忽略旧顶层字段。

第三步：修改 `save_resume_data()`，保存文件只写入：

```json
{
  "basics": {},
  "modules": []
}
```

第四步：修改 Markdown、HTML、PDF、AI 输入，全部直接读取 `modules`。

第五步：删除或隔离 `_legacy_from_modules()` 的运行时调用，只保留给测试或一次性迁移脚本使用。

## 9. AI 优化工作流

当前保持顺序工作流，不引入正式 LangGraph。

调用链：

```text
app.py:run_ai_optimization()
  -> core/service.py:optimize_resume_by_jd()
  -> workflow/graph.py
  -> InfoCollectorAgent
  -> ResumeWriterAgent
  -> merge / fallback
  -> 回写 resume_document["resume"]
```

### 7.1 输入

```python
{
    "jd_text": document["inputs"]["jd_text"],
    "current_resume": document["resume"],
    "uploaded_files": document["inputs"]["uploaded_files"],
    "style_params": document["style"]
}
```

### 7.2 输出

输出必须是合法简历 JSON，并通过 `ensure_resume_shape()` 归一化。

v3.0 的目标输出格式必须是标准 `basics + modules`：

```json
{
  "basics": {},
  "modules": []
}
```

AI 不应继续返回旧顶层字段作为主结果。

成功后：

- 更新 `resume_document["resume"]`
- 更新 `runtime.workflow_logs`
- 调用 `notify_document_changed(..., event="force")`
- 保存到 `data/resume_data.json`

失败时：

- API Key 缺失：提示配置 `.env`
- JSON 解析失败：提示重试或更换模型
- 其他异常：显示异常信息

### 7.3 AI 合并策略

当前 `merge_resume()` 只合并部分旧顶层字段，风险是：

- `education` 不会被 AI 更新。
- `basics` 不会被 AI 更新。
- `custom` 模块不会被 AI 更新。
- AI 返回的标准 `modules` 可能被忽略。

推荐改造为 modules 级合并：

```python
def merge_resume_modules(original: dict, optimized: dict) -> dict:
    original = ensure_resume_shape(original)
    optimized = ensure_resume_shape(optimized)

    original_by_id = {m["id"]: m for m in original["modules"]}
    merged_modules = []

    for optimized_module in optimized["modules"]:
        module_id = optimized_module.get("id")
        if module_id in original_by_id:
            base = original_by_id[module_id]
            merged = {
                **base,
                "title": optimized_module.get("title", base.get("title")),
                "visible": optimized_module.get("visible", base.get("visible", True)),
                "order": optimized_module.get("order", base.get("order", 0)),
                "content": optimized_module.get("content", base.get("content", {})),
            }
            merged_modules.append(merged)
        else:
            merged_modules.append(optimized_module)

    return {
        "basics": optimized.get("basics", original.get("basics", {})),
        "modules": merged_modules,
    }
```

如果担心 AI 修改事实字段，可以在 Prompt 中约束：

- 不改姓名、联系方式、学校、公司、时间等事实字段。
- 主要优化描述、亮点、技能表达、自我评价。
- 保留模块 ID 和模块类型。

## 10. 渲染实现

### 8.1 HTML

文件：

```text
renderers/html_renderer.py
```

职责：

- 合并模板样式。
- 根据 `modules` 的 `visible`、`order` 和 `has_module_content()` 渲染。
- 根据模块类型选择渲染方式。
- 输出完整 HTML 文档。

关键函数：

```python
build_style_params(style_params)
render_resume_preview_html(resume_data, style_params)
build_full_preview_document(resume_data, style_params)
```

### 8.2 Markdown

文件：

```text
renderers/markdown_renderer.py
core/markdown.py
```

职责：

- 基于当前简历数据生成 Markdown。
- 遵守模块可见性、内容判定和排序。
- 新逻辑必须直接从 `modules` 读取，不再从旧顶层字段读取。

### 8.3 PDF

文件：

```text
renderers/pdf_renderer.py
```

当前方案：

```text
HTML 预览 -> Playwright 截图 PNG -> Pillow 转 PDF
```

技术约束：

- PDF 视觉接近 HTML 预览。
- PDF 偏图片化，不保证文本可搜索或可复制。
- PDF 生成时强制 `preview_scale = 100`，避免预览缩放影响导出尺寸。
- 截图目标固定为 `.resume-a4`，避免截到整个页面背景。

### 8.4 PDF 风险说明

当前 PDF 是图片化 PDF，适合演示和视觉一致性，但不适合真实投递：

- 文本不可选择。
- 文本不可搜索。
- ATS 或招聘系统可能无法解析。
- 文件体积可能更大。

正式投递版本建议改为文本型 PDF：

```text
方案 A：Playwright page.pdf()
方案 B：WeasyPrint
方案 C：ReportLab 原生绘制
```

v3.0 当前不立即切换方案，但必须在界面或文档中明确“当前 PDF 为视觉优先截图方案”。如果项目目标转向真实投递，PDF 技术路线应优先调整。

## 11. 样式系统

样式配置来源：

```text
config.json -> resume_style.defaults / resume_style.templates
```

当前模板：

- `modern_blue`
- `elegant_gray`
- `emerald_pro`

后续模板不应只改变颜色，应允许布局结构不同：

- `classic_compact`：黑白、少卡片、少阴影，适合正式投递和 ATS。
- `modern_clean`：轻量强调线，适合互联网岗位。
- `two_column`：左侧信息栏 + 右侧经历，适合内容多的人。
- `student_minimal`：教育、项目、技能优先，适合校招。

投递型简历应减少大圆角、阴影、渐变和过度标签化展示，提高信息密度。

样式参数通过 `build_style_params()` 合并：

1. 默认样式。
2. 当前模板 preset。
3. 用户在 Sidebar 中的控件值。
4. 运行态预览缩放。

Sidebar 控件：

- 模板主题
- 正文字号
- 页面边距
- 行高
- 显示头像
- 重置为推荐样式

保存策略：

- 持久化样式时移除 `preview_scale`。
- `preview_scale` 属于运行态，不写入 `config.json`。

## 13. 通用模块结构化

当前新增模块统一为 `custom`，如果内容仅是字符串列表，后续会有两个问题：

- AI 无法判断标题、时间、描述、亮点之间的关系。
- 渲染器无法针对证书、获奖、论文、开源项目等内容做更好的排版。

因此 v3.0 推荐将 `custom` 的 item 从字符串升级为结构化对象：

```json
{
  "title": "全国数学建模竞赛一等奖",
  "subtitle": "团队负责人",
  "date": "2024",
  "description": "负责建模、代码实现和论文撰写。",
  "highlights": [
    "完成数据清洗与模型验证",
    "论文获得省级一等奖"
  ]
}
```

### 11.1 兼容规则

旧字符串：

```json
"英语六级"
```

迁移为：

```json
{
  "title": "英语六级",
  "subtitle": "",
  "date": "",
  "description": "",
  "highlights": []
}
```

### 11.2 UI 编辑规则

通用模块编辑器至少提供：

- 标题
- 副标题
- 时间
- 描述
- 亮点，每行一条

如果界面需要保持简洁，可以默认只展示标题和描述，将副标题、时间、亮点放入可展开区域。

## 16. 高风险问题与收敛方案

| 风险 | 当前表现 | v3.0 收敛方案 |
|------|----------|----------------|
| 双内容源互相覆盖 | `modules` 和旧顶层字段互相同步 | 只保留 `basics + modules`，旧字段只用于导入迁移 |
| 空模块进入导出 | 只判断 `visible`，空项目显示“暂无” | 增加 `has_module_content()`，HTML/Markdown/PDF 统一过滤 |
| AI 合并范围偏窄 | 只合并部分旧顶层字段 | AI 返回标准 `modules`，按模块 ID 合并 |
| custom 表达能力弱 | 每行字符串 | 升级为结构化 item |
| PDF 不适合真实投递 | 截图转 PDF | 当前明确标注风险，正式投递版本切到文本 PDF |
| PDF 阻塞页面 | download_button data 自动触发生成 | 改为先点击生成，成功后再下载 |
| 每轮 rerun 写磁盘 | Sidebar 末尾自动保存 | 改为 dirty 工作副本 + 显式保存 |
| 预览重复生成 | 每次 render 都 build HTML | 使用 `runtime.preview_html` 签名缓存 |

## 18. 后续维护原则

1. 新增 UI 控件时，优先让控件写入 `resume_document` 对应节点。
2. 内容源只允许 `basics + modules`。
3. 旧顶层字段只能在导入迁移阶段使用。
4. 不要在 widget 实例化后回写相同 key 的 `st.session_state`。
5. 新增导出格式时，作为 `resume_document` 的派生结果实现。
6. 新增模板时，只更新 `config.json` 和渲染器样式映射，避免改业务流程。
7. AI 输出必须经过 `ensure_resume_shape()`，且主结构必须是 `modules`。
8. 所有模块展示逻辑必须遵守 `visible`、`order` 和 `has_module_content()`。
9. PDF、Markdown、HTML 必须消费同一份简历 JSON，避免三套输出不一致。
10. PDF 生成必须由用户显式触发，不得阻塞页面 rerun。
11. 普通输入框编辑不得自动写磁盘。

## 19. 验收清单

- 应用可通过 `streamlit run app.py` 启动。
- Sidebar 编辑基础信息后，右侧预览同步。
- 模块新增、删除、上移、下移、显示隐藏可用。
- 三套模板可切换。
- 字号、边距、行高、头像开关影响预览。
- Markdown 下载内容与当前模块一致。
- PDF 下载内容与当前预览一致。
- 修改输入框不会自动触发 PDF 生成。
- 修改输入框不会立即写 `data/resume_data.json`。
- 点击“生成 PDF”后才显示可下载 PDF。
- 预览 HTML 在内容签名不变时复用缓存。
- 空模块不会进入 HTML、Markdown、PDF。
- 保存后重新加载，不会因为旧顶层字段覆盖 `modules` 导致内容回退。
- AI 优化后 custom、education、basics 不会因为旧合并逻辑丢失。
- PDF 生成后，继续修改内容会清空旧 PDF 缓存并重新生成。
- 空 JD 可执行通用 AI 优化或本地兜底。
- 有 JD 可执行针对性 AI 优化或本地兜底。
- 撤销上一次 AI 优化可用。
- 不出现 Streamlit widget key 被实例化后回写的异常。

## 20. 结论

v3.0 的核心不是增加功能，而是把当前功能收敛到稳定的数据流：

```text
统一 JSON 文档（basics + modules）
  -> 观察者同步
  -> HTML 预览 / Markdown / PDF
  -> AI 优化回写同一 JSON
```

后续开发只要遵守这个数据流，当前项目可以继续保持轻量、可演示、易教学，同时避免预览、Sidebar、PDF 各自维护状态导致的不一致问题。
