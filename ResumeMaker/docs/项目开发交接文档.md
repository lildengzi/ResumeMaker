# 智能简历生成器 - 项目开发交接文档

> 当前文档用于交接项目真实状态。历史规划和扩展想法请看 `docs/开发文档.md`、`docs/开发文档v2.0.md`、`docs/MVP_开发文档.md`，但后续开发判断应以本文档、`README.md` 和实际代码为准。

---

## 1. 当前项目定位

当前项目是一个 **Streamlit 学习项目**，不是完整商业级产品。它已经具备可演示的中文简历生成流程：

- 左侧编辑简历内容和样式
- 右侧 A4 风格 HTML 预览
- 可调用 OpenAI 兼容 LLM 根据 JD 优化简历
- LLM 不可用时有本地兜底生成
- 可导出 Markdown 和 PDF
- 可本地保存到 `data/resume_data.json`

当前阶段重点是保持项目可运行、结构清晰、适合继续教学式迭代。

---

## 2. 当前已实现能力

### 2.1 简历编辑

- 基础信息编辑
- 教育经历、技能、项目经历、工作/实习经历、校园经历、自我评价编辑
- 基础信息使用个人网站和作品集字段，旧数据中的 GitHub 会迁移到个人网站
- 个人照片上传，上传目录为 `data/uploads/`
- 模块新增、删除、排序、显示/隐藏、标题编辑
- 新增模块统一创建通用模块，用户通过标题和内容定义用途

### 2.2 AI 优化

- 支持填写 JD 文本
- 支持“智能生成简历”
- 工作流会按顺序执行信息收集 Agent 和简历撰写 Agent
- AI 优化结果会直接回写到左侧编辑面板，用户继续在 Sidebar 中手动修改
- 支持撤销上一次 AI 优化
- API Key 缺失、LLM 调用失败或输出不可解析时，走本地兜底逻辑

### 2.3 预览与导出

- 右侧使用 HTML A4 风格预览
- 支持 3 套主题：
  - `modern_blue`
  - `elegant_gray`
  - `emerald_pro`
- 支持字号、页面边距、行高、紧凑布局、头像显示等样式调节
- Markdown 下载入口在 Sidebar
- PDF 下载入口在右侧预览区顶部
- PDF 生成实现在 `renderers/pdf_renderer.py`

### 2.4 工程结构

- `core/`：数据、LLM、Markdown、资源处理、服务编排
- `agents/`：信息收集 Agent、简历撰写 Agent、AgentFactory
- `workflow/graph.py`：顺序工作流编排
- `renderers/`：HTML / Markdown / PDF 渲染
- `ui/`：Sidebar、Preview
- `tools/`：OCR、URL、上传文件解析等扩展接口
- `tests/`：当前最小测试集

## 3. 关键目录与文件

```text
app.py
```

Streamlit 主入口，负责初始化 `st.session_state`、渲染左右区域、触发 AI 优化、保存、撤销、导出和模块管理。

```text
config.py
config.json
```

配置中心。`config.json` 管理默认配置、LLM 参数、存储路径、预览参数、Prompt 和兜底文案；`config.py` 负责加载配置并合并环境变量。

```text
smart_resume_core.py
```

兼容入口层。历史代码可继续从这里导入公共能力，新代码优先直接使用 `core/`。

```text
core/data.py
```

负责默认数据、数据归一化、`data/resume_data.json` 读写、简历合并、旧字段与 `modules` 结构兼容。

```text
core/service.py
```

负责本地兜底和工作流入口封装，是 `optimize_resume_by_jd()` 的实际编排层。

```text
workflow/graph.py
```

当前是轻量顺序工作流，不是真 LangGraph。执行顺序为：

```text
InfoCollectorAgent -> ResumeWriterAgent
```

```text
renderers/html_renderer.py
```

负责 HTML 预览文档生成、主题、样式参数合并和模块渲染。

```text
renderers/markdown_renderer.py
core/markdown.py
```

负责 Markdown 导出。当前仍有旧字段兼容逻辑，后续建议进一步统一到 `modules`。

```text
renderers/pdf_renderer.py
```

负责 PDF 导出。当前链路是：

```text
HTML 预览 -> Playwright 截图 PNG -> Pillow 转 PDF
```

这不是 ReportLab 方案，也不是文本可搜索的 PDF 优先方案。

```text
tools/
```

当前是基础骨架：

- `permission.py`：限制工作区路径访问
- `file_tools.py`：上传文件扫描、Markdown/txt 基础读取，PDF 结构化解析未完成
- `web_tool.py`：基础网页文本抓取接口，不是生产级正文抽取
- `ocr_tool.py`：OCR 占位，未接入真实 OCR 引擎

## 4. 当前不是哪些东西

为避免后续误解，当前项目明确不是：

- 不是 FastAPI 前后端分离项目
- 不是 Docker 化部署项目
- 不是生产级招聘/简历 SaaS
- 不是正式 LangGraph 工作流
- 不是完整 OCR 系统
- 不是完整旧简历解析系统
- 不包含评分 Agent、面试 Agent 或多版本简历管理

这些内容可能出现在历史开发文档里，但当前阶段不作为优先目标。

## 5. 已知问题

### 5.1 Python 版本兼容风险

在 Python 3.14 环境下，`langchain_core` / `pydantic.v1` 可能出现兼容性警告。建议使用 Python 3.11 或 3.12 作为主要开发环境。

### 5.2 Streamlit 环境噪声

部分环境中运行 `streamlit run app.py` 时，可能因为已安装的 `transformers` / `torchvision` 相关包产生 watcher 噪声日志。通常不影响主流程，但会污染终端输出。

### 5.3 PDF 是图片化方案

当前 PDF 视觉上接近 HTML 预览，但本质上是截图转 PDF：

- 优点：预览和导出视觉更一致
- 缺点：文本不可搜索/复制，文件体积可能偏大

后续需要在“视觉优先”和“文本 PDF 优先”之间明确取舍。

### 5.4 Markdown / PDF 数据源仍需统一

当前 `core.data.ensure_resume_shape()` 维护 `modules` 与旧顶层字段的兼容，因此 Markdown / PDF 可继续工作。但长期看应让 HTML / Markdown / PDF 都直接基于 `modules` 渲染，避免双结构维护。

### 5.5 Tools 层测试不足

`tools/permission.py`、`tools/file_tools.py`、`tools/web_tool.py` 当前缺少足够测试。后续扩展 OCR、URL 抓取和旧简历解析前，建议先补基础测试。

### 5.6 模块管理体验仍可优化

新增、删除、排序和显示/隐藏已具备，但仍可优化：

- 新增模块后自动展开
- 删除二次确认
- 复制模块
- 空模块提示与校验

## 6. 推荐验收清单

下一轮编码前建议先做一次人工验收，形成当前版本基线：

- 应用能通过 `streamlit run app.py` 启动
- 3 套模板都能正常切换
- 字号、边距、行高、紧凑布局、头像显示开关能影响预览
- Sidebar 编辑后右侧预览同步
- 新增模块、删除模块、上移/下移模块不再触发重复 key 错误
- Markdown 能正常下载
- PDF 能正常下载
- 空 JD 与有 JD 两种场景都能完成智能生成或本地兜底
- 撤销上一次 AI 生成可用

## 7. 下一步开发建议

### 7.1 第一优先级：建立稳定基线

- 修复或记录本机 Python/pytest 环境问题
- 补 `tools/` 层最小测试
- 做一轮人工验收，记录 3 到 5 个最影响演示的问题

### 7.2 第二优先级：统一渲染数据源

- 让 Markdown 直接基于 `modules` 渲染
- 让 PDF 直接基于 `modules` 渲染
- 确保 HTML / Markdown / PDF 使用同一套模块排序、可见性、标题和内容规则

### 7.3 第三优先级：明确 PDF 策略

二选一：

- 保持当前截图转 PDF：文档中明确“视觉优先，图片化 PDF”
- 改为 Playwright 原生 `page.pdf()`：更接近真正 HTML-to-PDF，便于后续支持可搜索文本

### 7.4 第四优先级：增强已有简历解析

暂不接真实 OCR。先增强已有简历解析更实用：

- Markdown / txt 读取后生成结构化草稿
- PDF 先做文本抽取，再做结构化候选内容
- 让用户确认后再合并到 `resume_data`

## 8. 暂不建议投入的方向

当前阶段暂不建议优先做：

- FastAPI 改造
- Dockerfile / 部署方案
- 评分 Agent
- 面试 Agent
- 真实 OCR 引擎接入
- 正式 LangGraph 改造

这些方向不是不能做，而是会增加依赖、复杂度和验证成本。当前更需要先稳住可运行体验、测试基线和渲染一致性。

## 10. 运行方式

安装依赖：

```bash
pip install -r requirements.txt
```

首次使用 PDF 导出前安装 Chromium：

```bash
python -m playwright install chromium
```

启动应用：

```bash
streamlit run app.py
```

运行测试：

```bash
pytest
```

LLM 配置写入 `.env`：

```env
OPENAI_API_KEY=your_api_key
OPENAI_BASE_URL=
LLM_MODEL=gpt-4o-mini
LLM_TEMPERATURE=0.2
```
